# 会话不中断专项自查（执行记录）

本文档列出我将依次完成的检查项，逐步确认当前代码是否具备前述防断线能力；每一项会写明目的、检查方式、预期结果，以及是否需要进一步行动。未动任何代码，仅规划与记录检查步骤。

## 1. PWA 唤醒与自愈链路
- **目的**：确认切后台/回前台时会触发 usePWAKeeper 的唤醒逻辑，包含 removeAllChannels + getSession 与必要时的硬刷新。
- **实测结论**：❌ 未通过（iphone16 Pro Max，iOS 26.2，PWA 安装版）。
  - 复现：外部停留 ~5s 回前台，页面元素仍在，但评论展开为空，发布记忆一直转圈，所有操作都在 loading。
  - 表现推测：回前台未触发有效的会话刷新或连接重建，导致接口请求被 401/超时但 UI 保留旧缓存。
- **初步定位方向**：
  1) 前台可见性切换时立即执行 refreshSession（而不是仅 getSession）并强制重拉核心数据；
  2) 回前台时对所有待用的 Supabase channel 做 removeAllChannels 后重建，防止假连接；
  3) 将唤醒硬刷新阈值从 5 分钟下调到几秒/按需策略，至少在 PWA 上增加一次「可见性切换即强制 refresh」兜底；
  4) 为回前台的首个请求增加 4-8 秒超时 + 明确报错提示，避免静默卡圈。
- **下一步动作（待实现）**：
  - 在 App.tsx 的可见性回调里添加“可见即 refreshSession + fetch 核心数据 + 重建 channel”的快速路径。
  - 增加前台恢复时的接口超时与错误提示，避免无感失败。
  - 视情况把 HARD_RELOAD_THRESHOLD_MS 下调或允许 iOS PWA 配置更激进的硬刷新。

## 2. 前台恢复数据重拉（rehydrateIfEmpty）
- **目的**：确认回前台时会话刷新和核心数据补拉运行正常。
- **做法**：
  1) 正常登录后，切后台 1-2 分钟，回前台。
  2) 检查记忆列表、好友、账本是否重新请求（可看 Loading 或数据闪动；若有 Proxy 工具可抓包）。
  3) 若本地缓存被系统清掉，确认是否自动用缓存/网络恢复，界面不为空白。
- **预期**：回前台后数据保持或迅速恢复；无“空列表+操作失败”。
- **后续**：若无请求发出，排查 lastRefreshRef 节流逻辑或 shouldAllowRefresh 配置。

## 3. 会话失效事件捕获与 UI 提示
- **目的**：验证 sessionInvalid 流程能被触发并在 UI 弹出红色“重新连接”提示。
- **做法**：
  1) 手动让 token 失效（例如在另一端登出，或调试工具清除 supabase 存储）。
  2) 触发任意需要鉴权的操作（刷新列表/发表评论），观察是否出现底部红色条提示。
- **预期**：出现红条提示；未直接悄悄报错卡住。
- **后续**：若无提示，检查 ORBIT_AUTH_INVALID_EVENT 是否被触发，或拦截器未正确派发。

## 4. 一键“重新连接”按钮行为
- **目的**：确认 attemptReconnect 能成功刷新会话并刷新数据，失败时能强制登出。
- **做法**：
  1) 在出现红条时点击“重新连接”。
  2) 期待调用 refreshSession → 拉取 profile → 拉取记忆/好友/账本；若失败则自动 signOut 并弹登录。
- **预期**：按钮可恢复正常操作；若网络/会话确实不可用，会跳回登录而不是假死。
- **后续**：若按钮无效，需检查 supabase.auth.refreshSession 返回值与错误分支。

## 5. 登录弹窗时机与脏数据隔离
- **目的**：确认 allowAuthModal 延时、未登录强制弹窗，以及切换账户后的数据清理。
- **做法**：
  1) 退出登录后重进 App，确认闪屏结束后才弹登录，不阻塞首屏渲染；
  2) 用账号 A 登录后退出，再登录账号 B，检查记忆/好友/账本是否被清空后再拉取（无前一账号残留）。
- **预期**：登录弹窗时机合理；账号切换无脏数据穿透。
- **后续**：若残留，需检查 resetClientData 调用链。

## 6. 演示模式隔离
- **目的**：确保 demo 模式不会污染真实会话，并可正常退出。
- **做法**：
  1) 进入 demo（query 参数 demo=1），检查顶部演示横幅；
  2) 退出演示后，应清空 demo 数据并回到登录；
  3) 重新登录真实账户，确认无演示数据残留。
- **预期**：数据隔离，退出后恢复干净状态。
- **后续**：若混入，排查退出演示的清理逻辑。

## 7. 自动刷新与缺失恢复
- **目的**：确认定时与可视性触发的轻量刷新在弱网时不会假死。
- **做法**：
  1) 保持前台 1 分钟以上，观察是否有自动刷新（定时 60s）。
  2) 人为清空内存中的 memories/friends（开发者工具）后触发 visibilitychange，验证是否补拉。
- **预期**：自动刷新生效，缺失数据会被补齐。
- **后续**：若无刷新，检查 lastAutoRefreshRef 节流与 shouldAllowRefresh。

## 8. UI 体验与主题保障
- **目的**：确认主题/字体在首屏已生效，避免因闪屏样式抖动被误判为“卡顿/断线”。
- **做法**：首屏加载时观察字体与主题是否立即应用；若出现白屏或对比度突变，记录具体机型与复现步骤。
- **预期**：无明显闪烁，深/浅色正确。
